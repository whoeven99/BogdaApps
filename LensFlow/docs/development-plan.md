# 开发文档与执行计划

- 版本：`V1.2`
- 文档类型：实施路线图与研发执行文档
- 关联文档：`docs/prd.md`、`docs/design.md`、`docs/agent.md`

本文档基于 `docs/prd.md` 的 `V1.5` 范围定义，并联合 `docs/design.md` 与 `docs/agent.md` 作为当前版本的研发执行文档，覆盖目标架构、模块拆解、依赖关系、交付物、接口顺序、测试策略与当前推进路线。

## 0. 执行总览

### 0.1 实施目标

- 将当前演示型推荐与零散规则能力，升级为可运营、可配置、可下单的 LensFlow 正式产品。
- 形成从后台配置、前台智能导购、库存命中、Bundle 结算到订阅闭环的完整链路。
- 保证所有实现同时满足 `prd.md` 的业务范围、`design.md` 的交互规范、`agent.md` 的协作约束。

### 0.2 总体推进顺序

- 先完成底层参数、模板、规则与库存映射模型。
- 再完成后台配置入口与管理页。
- 再完成前台导购容器、状态机与匹配接口。
- 再打通普通订单与 Bundle 购买闭环。
- 最后推进订阅合同、周期订单与运营提醒。

### 0.3 当前核心风险

- 参数模型、动态规则、库存映射若设计不稳，会导致前后台配置和前台导购脱节。
- 主商品绑定关系和镜片知识规则若未单独建模，后续很容易退化成硬编码判断。
- 若前台仍以演示表单方式推进，会与 `design.md` 的任务型入口和交互状态机冲突。
- 若先做订单闭环再补 Bundle 和订阅约束，后续回改成本高。
- 非标准主题接入、变体同步、草稿恢复、知识规则解释展示都属于高风险集成项，需要提前纳入阶段设计。

## 1. 开发原则

- 先建立参数模型、单位模型和库存映射模型，再做前台插件和订单闭环。
- 后台商品配置、前台参数输入、订单创建、订阅合同流程必须共用同一套参数定义。
- 主商品绑定关系、镜片知识规则、库存规则和订单摘要必须使用同一套可追溯标识体系。
- 标准参数与自定义参数并存，避免把镜片或隐形眼镜行业逻辑硬编码在前台脚本中。
- `Theme App Extension` 通过 `App Proxy` 获取参数模板、匹配结果和下单结果，减少对主题模板的侵入。
- 前台导购必须遵循 `docs/design.md` 中的任务型入口、卡片式决策、摘要区、异常态与状态保持规范，不能退化为纯表单。
- 文档、设计、开发、测试与实施中的 Agent 协作，需遵循 `docs/agent.md` 的职责边界与输出标准。
- 订阅购买按“两阶段”推进：
  - 阶段一：订阅商品、订阅方案、首次下单传 `selling_plan`
  - 阶段二：订阅合同、周期订单、提醒与失败补偿
- 每完成一个业务闭环，都需要同步补测试、类型检查和 Shopify 相关校验。

## 2. 实施工作流

### 2.1 文档流

- 需求进入前先确认 `prd.md` 是否已定义业务边界。
- 涉及商品页操作、状态变化、页面反馈时，必须同步参考 `design.md`。
- 涉及自动化执行、回归方式、联调流程时，必须同步参考 `agent.md`。

### 2.2 研发流

- 阶段一：完成数据模型、主商品绑定与服务层能力。
- 阶段二：完成后台管理页与配置闭环。
- 阶段三：完成前台导购与购买链路。
- 阶段四：完成订阅、提醒、诊断与实施支持。

### 2.3 验证流

- 每个模块先完成类型定义与服务层测试。
- 页面改动必须验证主流程、异常流、恢复流。
- 接口改动必须验证 Shopify 上下文、权限、Metafield 与变体同步影响。
- 阶段收尾必须完成 `npm test`、`npm run typecheck`、`npm run build`。

## 3. 目标架构

### 3.1 后台应用

- 技术栈：`Shopify App + React Router + Polaris + Prisma`
- 能力范围：
  - 商品管理
  - 参数模板管理
  - 参数单位规格管理
  - 商品参数配置
  - 参数组合与库存映射
  - 订阅商品与订阅方案管理
  - 订单与合同跟踪

### 3.2 业务层

- 商品与 Metafield 读取：`app/services/shopify-products.server.ts`
- 参数推荐与匹配基础：`src/services/prescriptionRecommendations.ts`
- 前台聚合：`app/lib/storefront-lens.server.ts`
- 订阅方案与 Selling Plan：`app/services/subscription-plans.server.ts`
- 订阅诊断与状态汇总：`app/services/subscription-diagnostics.server.ts`
- 后续新增：
  - 参数模板服务
  - 主商品绑定服务
  - 专业知识规则服务
  - 单位规格服务
  - 参数组合映射服务
  - 订单创建服务
  - 订阅合同服务

### 3.3 前台接入

- 主题扩展：`extensions/lens-product-widget`
- 数据出口：
  - `/apps/product-match`
  - `/apps/create-parameterized-order`
  - `/apps/create-subscription-contract`
- 前台 JS：
  - 渲染任务型入口、商品优先入口、步骤容器和悬浮摘要
  - 按参数模板与流程配置动态生成步骤
  - 调用知识规则解析、导购规则解析与匹配接口
  - 调用普通订单和订阅下单接口
  - 处理草稿保存、恢复和变体切换

## 4. 工作流拆解

### 4.1 工作流 A：参数与规则底座

目标：建立可复用的参数、模板、单位、动态规则、知识规则和商品级覆盖基础能力。

核心内容：

- 参数定义模型
- 单位规格模型
- 模板与商品级覆盖模型
- 动态显隐与跳步规则模型
- 专业知识规则模型
- 主商品绑定模型
- 参数标准化与签名能力

前置依赖：

- 无

后置产出：

- 后台配置与前台导购共用的数据结构
- 匹配服务可直接消费的参数签名
- 知识规则与订单摘要可复用的规则标识

验收口径：

- 能表达标准参数、自定义参数、单位、步进、必填与商品级覆盖
- 能表达前台显隐、跳步、知识解释和推荐相关规则

### 4.2 工作流 B：库存与商品映射

目标：建立参数组合到真实货品、SKU、库存状态和组合可售性的稳定映射。

核心内容：

- 参数组合签名
- Shopify Variant 绑定
- 主商品与镜片绑定范围
- 缺货与替代方案结果
- Bundle 主商品与配品关系

前置依赖：

- 工作流 A

后置产出：

- 匹配服务
- 加购服务
- 结果页可消费的推荐与缺货数据
- 主商品切换后可复算的导购入口与绑定结果

验收口径：

- 能根据参数命中唯一货品或稳定返回明确错误/替代结果
- 能支持 Bundle 中主商品与配品的联动关系
- 能支持主商品、镜片 SKU、组合可售性三层校验

### 4.3 工作流 C：后台运营配置

目标：让商家能真正配置产品，而不是依赖硬编码。

核心内容：

- 参数模板管理页
- 单位规格管理页
- 商品参数配置页
- 专业规则页
- 库存映射页
- 导购流配置页
- Bundle 配置页

前置依赖：

- 工作流 A
- 工作流 B 的基础映射结构

后置产出：

- 商家可直接使用的配置后台

验收口径：

- 商家无需改代码即可完成参数模板、主商品绑定、专业规则、库存映射与导购流配置

### 4.4 工作流 D：前台导购体验

目标：将前台从演示型参数表单升级为正式导购流。

核心内容：

- 任务型入口
- 商品优先入口
- 步骤式导购容器
- 卡片选择与结果对比
- 悬浮摘要与进度反馈
- 草稿保存与恢复
- 阻断态与异常恢复
- 专业知识解释卡与替代建议

前置依赖：

- 工作流 A
- 工作流 B
- `design.md` 中的状态机与页面规范

后置产出：

- 商品页正式导购插件

验收口径：

- 前台不再直接暴露纯表单
- 双路径加商品优先导购与状态恢复可实际运行

### 4.5 工作流 E：普通订单与 Bundle 闭环

目标：打通从匹配结果到购物车、订单摘要与履约信息的完整链路。

核心内容：

- 匹配结果转购物车项
- Bundle 一并加购
- 参数摘要写入订单属性
- 最终确认页
- 主商品删除联动配品删除

前置依赖：

- 工作流 B
- 工作流 D

后置产出：

- 正式可售的一次性购买闭环

验收口径：

- 用户可从导购流顺利完成 Bundle 加购
- 商家可在订单中查看摘要与履约信息

### 4.6 工作流 F：订阅与运营闭环

目标：打通订阅商品、首单、合同、周期订单、提醒和诊断能力。

核心内容：

- 订阅商品标记与管理
- 前台订阅入口
- 抛期与订阅周期映射
- 折扣与建议购买量计算
- `selling_plan` 传递
- 订阅合同创建与回查
- 周期订单与提醒
- 订阅诊断与状态跟踪

前置依赖：

- 工作流 C
- 工作流 D
- 工作流 E

后置产出：

- 正式可运营的订阅链路

验收口径：

- 首次订阅下单能创建真实合同
- 后续周期订单与提醒链路可追踪
- 抛期变化与参数变化可触发重新确认
