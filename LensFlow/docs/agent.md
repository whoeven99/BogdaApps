# LensFlow Agent 协作规范

- 版本：`V1.0`
- 文档类型：Agent 协作与执行规范
- 适用对象：产品、设计、研发、测试、实施、运营、AI Agent
- 关联文档：`docs/prd.md`、`docs/design.md`

## 1. 文档目标

- 定义 AI Agent 在 LensFlow 项目中的职责边界与交付标准。
- 确保 Agent 的输出与 `prd.md` 的业务规则、`design.md` 的交互规范保持一致。
- 降低 Agent 生成内容与真实实现之间的偏差、遗漏和误操作风险。

## 2. Agent 角色定义

### 2.1 产品 Agent

- 将业务想法整理为需求项、待确认问题、范围边界和验收标准。
- 检查新增需求是否影响双路径导购、动态规则、Bundle 或订阅流程。

### 2.2 设计 Agent

- 将需求转成页面结构、交互状态、组件清单和异常流。
- 检查方案是否退化为纯表单，是否符合任务型入口与卡片式导购原则。

### 2.3 研发 Agent

- 将需求拆解为数据模型、接口、页面、服务与测试任务。
- 实现时必须复用已有参数模型、规则引擎、库存映射和订阅能力，避免重复造轮子。

### 2.4 测试 Agent

- 基于 PRD 与设计规范生成测试清单、边界场景和回归范围。
- 重点覆盖动态显隐、跳步、Bundle 联动、状态恢复和订阅分支。

### 2.5 实施 Agent

- 输出主题接入、手动安装、联调检查和上线核对清单。
- 重点关注非标准主题、变体同步失败、App Proxy 兼容和前台容器差异。

## 3. 协作原则

- 业务定义以 `prd.md` 为准。
- 交互与视觉行为以 `design.md` 为准。
- Agent 不得自行创造与上述文档冲突的业务规则。
- 遇到规则不明确、流程冲突或缺少前置条件时，Agent 应先提出澄清，而不是擅自补全关键业务逻辑。
- 涉及数据删除、订单影响、订阅合同影响的操作时，Agent 必须优先采用保守策略。

## 4. 标准工作流

### 4.1 需求型任务

- 读取 `prd.md`
- 定位影响章节
- 提炼范围变化、风险点、待确认问题
- 如涉及交互，联动更新 `design.md`
- 如涉及执行方法，联动更新 `agent.md`

### 4.2 设计型任务

- 读取 `prd.md` 中对应业务场景与验收标准
- 读取 `design.md` 中对应页面、状态机和组件规范
- 输出线框、交互说明、异常流和设计验收清单

### 4.3 开发型任务

- 先确认 PRD 范围与设计状态
- 再搜索现有实现、数据模型和接口
- 修改代码前说明计划和影响范围
- 修改后执行诊断、测试和必要验证

### 4.4 评审型任务

- 优先检查行为回归、规则偏差、状态遗漏、异常处理缺失
- 对照 `prd.md` 验证业务符合性
- 对照 `design.md` 验证交互一致性
- 对照 `agent.md` 验证交付流程与校验是否完整

## 5. 输入输出规范

### 5.1 输入要求

- 任务目标必须明确说明属于产品、设计、研发、测试或实施哪一类。
- 若任务涉及商品页交互，默认需要同时参考 `prd.md` 与 `design.md`。
- 若任务涉及流程自动化、批量修改、回归检查或实施协作，需同时参考 `agent.md`。

### 5.2 输出要求

- 输出必须标明影响对象，如文档、代码、接口、测试、页面。
- 输出必须标明是否影响：
  - 双路径导购
  - 动态显隐规则
  - Bundle 结算
  - 订阅链路
  - 手动安装或主题兼容
- 输出必须包含风险与未覆盖项，尤其是异常流和状态恢复。

## 6. 面向 LensFlow 的专项检查项

### 6.1 业务检查

- 是否同时考虑 `处方优先` 与 `功能优先`。
- 是否影响参数模板、规则引擎或库存映射。
- 是否改变前台隐藏 SKU、Bundle 删除联动或订阅购买入口。

### 6.2 交互检查

- 是否仍满足任务型入口，而不是退回为纯字段表单。
- 是否保留卡片选择、进度反馈、摘要区、结果对比和确认页。
- 是否覆盖移动端底部操作栏和桌面端摘要侧栏。

### 6.3 技术检查

- 是否影响 App Proxy、Theme App Extension、手动安装脚本。
- 是否影响 Shopify 商品变体同步、Metafield、订单属性、订阅合同。
- 是否需要同步更新测试与诊断逻辑。

## 7. 禁止事项

- 不得在缺少业务依据时擅自添加医学判断规则。
- 不得绕过既有导购流直接暴露隐藏 SKU 给消费者。
- 不得在未校验 Bundle 和订阅约束时直接拼接购物车逻辑。
- 不得忽略异常态，只覆盖正常下单路径。
- 不得在未知影响下覆盖或回滚用户已有改动。

## 8. 文档联动规则

- `prd.md` 有范围变更时，需检查是否同步更新 `design.md` 和 `agent.md`。
- `design.md` 有交互状态变更时，需检查是否影响 PRD 验收标准与测试口径。
- `agent.md` 有执行规范变更时，需检查是否影响文档、开发计划或测试流程。
- `development-plan.md` 应引用这三份文档的最新版本，作为执行落地基线。

## 11. Agent 验收标准

- 输出是否明确引用 `prd.md`、`design.md`、`agent.md` 中的相关约束。
- 输出是否说明影响范围与风险，而非只给结果。
- 输出是否覆盖正常流、异常流和恢复流。
- 输出是否避免与现有业务规则、交互规范和实施约束冲突。

## 12. 常用执行模板示例

### 12.1 文档更新任务示例

- 任务类型：产品
- 目标摘要：补充导购流中的异常恢复逻辑
- 影响文档：`prd.md`、`design.md`
- 影响范围：前台导购、摘要区、阻断态
- 是否影响双路径导购：是
- 是否影响动态显隐：是
- 是否影响 Bundle：否
- 是否影响订阅链路：可能
- 是否影响主题接入或手动安装：否
- 主要风险：需求与现有结果页逻辑冲突
- 待确认问题：草稿恢复是否跨设备同步

### 12.2 代码评审任务示例

- 评审对象：商品页导购插件改造
- 关联需求：任务型入口、卡片式选择、草稿恢复
- 是否符合任务型入口：待核验
- 是否存在纯表单退化风险：有，需要重点检查
- 是否保留卡片选择与方案对比：待核验
- 是否存在持续可见摘要：待核验
- 是否覆盖异常态与恢复流：需重点检查
- 移动端风险：底部操作栏可能被遮挡
- 桌面端风险：摘要栏与结果区冲突
- 结论：需结合页面实现与状态恢复逻辑进一步确认
